# 练习 06：测试架构

## 目标

学习如何组织较大的测试集，让测试既好写又好维护。

## 需求示例

- 用 fixture 准备共用数据
- 用参数化测试覆盖一组回归场景
- 用测试数据工厂生成可读性更好的测试数据
- 让失败信息能直接指向具体场景

## 建议

- 先从重复很多的测试数据开始整理
- 让测试意图比实现细节更重要
- 优先保证可读性，再考虑抽象复用
- 如果某个抽象让测试更难读，先退一步简化它

## 验收标准

完成这一练习后，你应该能回答：

- 哪些准备动作适合放进 fixture
- 哪些输入组合适合放进工厂
- 哪些场景适合参数化，哪些不适合
- 如何把回归用例写成一组可维护的命名场景

## 实战示例

- 阅读 `docs/chapter-6-example.md`
- 阅读 `docs/chapter-6-cli-e2e.md`
- 对照 `tests/test_chapter6.py` 理解 fixture、工厂和参数化的分工

## 讨论点

- 什么时候该用 fixture，什么时候不该用？
- 测试工厂会不会过度抽象？
- 命名场景和参数化列表，哪种更适合你的项目？
