# 第一章实战示例：为什么需要测试

这一章不先讲复杂概念，而是用一个很常见的工程问题来感受测试的价值：

> **满 100 元免运费**

这个规则看起来很简单，但实际上最容易出 bug 的地方正是 **边界条件**。

---

## 需求

- 订单金额小于 100 元：需要支付运费
- 订单金额大于等于 100 元：免运费
- 订单金额不能是负数

---

## 为什么这个例子适合入门

很多 bug 不是来自“完全不会写”，而是来自：

- 手工测试只测了 99 和 101，忘了 100
- 改需求时把 `>` 写成了 `>=`
- 逻辑看起来对，但边界条件漏掉了

测试可以把这种问题变成**可重复、可自动执行**的检查。

---

## 推荐的 TDD 顺序

### 1. 先写一个会失败的测试

先关注最关键的边界值：

```python
assert is_free_shipping(100) is True
```

如果实现还没写好，这条测试应该先失败。

### 2. 再写最少代码让它通过

实现函数时，只需要让规则成立：

```python
return order_amount >= threshold
```

### 3. 最后在测试保护下重构

当测试已经覆盖住行为后，你可以放心地：

- 提取常量
- 拆分函数
- 改善命名
- 增加说明文档

因为一旦不小心改坏，测试会立刻提醒你。

---

## 本章代码位置

- 实现：`src/tdd_learning_python/chapter1.py`
- 测试：`tests/test_chapter1.py`

---

## 你应该观察什么

1. `99 -> False`
2. `100 -> True`
3. `101 -> True`
4. `-1` 会抛出异常

这些测试覆盖的不是“代码细节”，而是**业务行为**。

如果你想看更接近真实使用场景的版本，可以继续阅读 `docs/chapter-1-cli-e2e.md`，那里会展示如何把这个规则包装成一个最小命令行程序，并用 E2E 测试模拟用户输入。

这就是测试最重要的价值：

- 降低回归风险
- 让你更敢改代码
- 让边界条件被自动保护起来
